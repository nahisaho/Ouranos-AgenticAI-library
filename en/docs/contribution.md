## 貢献の方法

### 貢献の種類

以下のような貢献を歓迎します:

1. **新しいプロンプトの追加**
   - 既存カテゴリへの新規プロンプト
   - 新しいカテゴリの提案と作成

2. **既存プロンプトの改善**
   - バグ修正
   - 内容の充実化
   - わかりやすさの向上

3. **ドキュメントの改善**
   - README、ガイドの更新
   - 誤字脱字の修正
   - 翻訳の追加

4. **フレームワークの追加**
   - 新しいフレームワークの参照集への追加
   - 既存フレームワークの説明改善

## プロンプト作成ガイドライン

### 基本要件

新しいプロンプトを作成する際は、以下の要件を満たしてください:

1. **YAMLフロントマターを含む**
   - 必須フィールド: `id`, `category`, `frameworks`, `dialogue_stages`, `version`, `tags`, `created`, `updated`

2. **必須セクションを含む**
   - 役割定義
   - 対話フロー
   - 活用する専門知識・フレームワーク
   - 出力形式

3. **推奨セクション**
   - 使用例
   - 注意事項
   - バージョン情報

### テンプレート構造

```markdown
---
id: [カテゴリ]-[具体的な名前]
category: [カテゴリID]
frameworks:
  - [フレームワーク1]
  - [フレームワーク2]
dialogue_stages: [数値]
version: 1.0.0
tags:
  - [タグ1]
  - [タグ2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

# [プロンプトタイトル]

## 役割
[AIの役割と専門性の定義]

## 対話フロー

### Stage 1: [ステージ名]
**目標**: [このステージの目標]

[質問項目や説明]

**Stage 1の出力**: [このステージの出力]

---

### Stage 2: [ステージ名]
[同様に記述]

---

## 活用する専門知識・フレームワーク

### [フレームワーク名]
[説明と適用方法]

---

## 出力形式
[最終的な出力のフォーマット]

---

## 使用例（オプション）
[具体例]

---

## バージョン情報
- **Version**: 1.0.0
- **Created**: YYYY-MM-DD
- **Updated**: YYYY-MM-DD
- **Status**: Active
```

### 命名規則

**ファイル名:**
- 形式: `{domain}-{specific-name}.md`
- 小文字とハイフンのみ使用
- 例: `business-strategy-consultant.md`

**プロンプトID:**
- ファイル名と同じ（拡張子なし）
- 例: `business-strategy-consultant`

**タグ:**
- 小文字、ハイフン区切り
- 具体的で検索可能なものを選択
- 3-5個程度が適切

### 品質基準

プロンプトは以下の基準を満たす必要があります:

1. **明確性**
   - 役割と目的が明確
   - 曖昧な表現を避ける

2. **構造化**
   - 論理的な流れ
   - セクションが適切に分割されている

3. **実用性**
   - 実際に使える内容
   - 具体的な質問項目や出力形式

4. **専門性**
   - 適切なフレームワークの活用
   - 専門知識の正確な記述

5. **一貫性**
   - 他のプロンプトとスタイルが統一
   - テンプレート構造に準拠

## コーディング規約

### マークダウンスタイル

1. **見出し**
   - `#`の後にスペースを入れる
   - 階層を適切に保つ

2. **リスト**
   - インデントは2スペース
   - リスト前後に空行を入れる

3. **コードブロック**
   - 言語指定を含める
   - ````markdown, ```yaml, ```bash など

4. **強調**
   - 太字: `**テキスト**`
   - イタリック: `*テキスト*`（必要最小限）

### YAMLスタイル

1. **インデント**
   - 2スペース

2. **リスト**
   - `-` の後にスペース

3. **文字列**
   - 引用符は必要な場合のみ

## プルリクエストのプロセス

### 1. フォークとクローン

```bash
# リポジトリをフォーク（GitHub上で）
# フォークしたリポジトリをクローン
git clone https://github.com/[あなたのユーザー名]/ouranos-agentic-library.git
cd ouranos-agentic-library
```

### 2. ブランチの作成

```bash
# 機能ブランチを作成
git checkout -b feature/your-feature-name

# 例:
git checkout -b feature/add-marketing-strategy-prompt
```

**ブランチ命名規則:**
- `feature/`: 新機能追加
- `fix/`: バグ修正
- `docs/`: ドキュメント更新
- `refactor/`: リファクタリング

### 3. 変更の作成

1. プロンプトファイルを作成または編集
2. `manifest.yml`を更新（新規プロンプトの場合）
3. 必要に応じてドキュメントを更新

### 4. manifest.ymlの更新

新しいプロンプトを追加した場合:

```yaml
prompts:
  - id: your-prompt-id
    category: category-name
    title: プロンプトタイトル
    file: prompts/category-name/your-prompt-id.md
    frameworks:
      - Framework 1
      - Framework 2
    tags:
      - tag1
      - tag2
    version: 1.0.0
    created: 2025-11-19
    updated: 2025-11-19
```

### 5. コミット

```bash
# 変更をステージング
git add .

# コミット（わかりやすいメッセージで）
git commit -m "Add: マーケティング戦略プロンプトを追加"
```

**コミットメッセージ規約:**
- `Add:` 新規追加
- `Update:` 更新
- `Fix:` バグ修正
- `Docs:` ドキュメント変更
- `Refactor:` リファクタリング

### 6. プッシュとプルリクエスト

```bash
# リモートにプッシュ
git push origin feature/your-feature-name
```

GitHub上でプルリクエストを作成:
1. 変更内容の説明を記入
2. 関連するIssueがあればリンク
3. レビュワーを割り当て（可能な場合）

### 7. レビューと修正

- レビューアからのフィードバックに対応
- 必要に応じて修正をコミット
- 議論を通じて改善

## プルリクエストのチェックリスト

プルリクエストを作成する前に確認:

- [ ] YAMLフロントマターが正しく記述されている
- [ ] 必須セクションがすべて含まれている
- [ ] `manifest.yml`が更新されている（新規プロンプトの場合）
- [ ] マークダウンの構文が正しい
- [ ] リンクが正しく機能する
- [ ] 誤字脱字がない
- [ ] 一貫したスタイルが保たれている
- [ ] 実際に使用してテストした

## コミュニティガイドライン

1. **尊重**
